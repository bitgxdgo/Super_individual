# Execution Stage

**目的**: 按计划逐步实现功能。

## 执行要求

现在**按计划**、**完整地**实现。

### 实施要求

- 编写优雅、最小、模块化的代码
- 严格遵守现有的代码模式和约定
- 包含全面、清晰的注释/文档，特别是复杂逻辑
- 在实施每个步骤时：
  - 使用相应的表情符号更新 markdown 跟踪文档
  - 更新整体进度百分比
  - 完成后勾选子任务

### 代码质量标准

- **优雅**: 代码应该干净、可读、易于理解
- **最小**: 不要过度工程化 - 只构建需要的东西
- **模块化**: 保持关注点分离
- **可测试**: 编写可以轻松测试的代码
- **文档化**: 添加注释解释"为什么"，而不仅仅是"什么"

### 实施流程

每个步骤：

1. **开始前**
   - 将步骤状态更新为 🟡 进行中
   - 审查步骤要求和受影响的文件
   - 考虑实施方法

2. **实施期间**
   - 遵循现有模式
   - 编写清晰的代码和注释
   - 在进行时勾选子任务
   - 如果发现问题或不明确之处，请立即提出

3. **完成后**
   - 将步骤状态更新为 ✅ 完成
   - 更新总体进度百分比
   - 测试实施
   - 简要总结完成的内容

### 进度更新格式

```markdown
**总体进度**: `33%`

### ✅ 步骤 1: [步骤名称]
[简要说明完成的内容]

### 🟡 步骤 2: [步骤名称]
[当前正在做什么]
- [x] 子任务 1
- [ ] 子任务 2

### ⬜ 步骤 3: [步骤名称]
[待办]
```

## 重要原则

1. **不要偏离计划**: 如果需要更改，先讨论
2. **增量测试**: 不要等到最后才测试
3. **保持沟通**: 报告进度和任何阻碍
4. **质量优于速度**: 正确完成比快速完成更重要

## 遇到问题时

如果在实施过程中遇到：
- 意外的技术挑战
- 不明确的需求
- 需要架构决策
- 潜在的范围变更

**立即暂停并讨论**，而不是自行假设。
