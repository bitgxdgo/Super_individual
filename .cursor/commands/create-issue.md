# Create Issue

**目的**: 在开发过程中快速捕获 bug/功能需求

## 使用场景
用户在开发过程中想到了一个 bug/功能需求

## 你的目标

创建一个完整的 issue，包含：
- 清晰的标题
- TL;DR - 简要说明这个 issue 的内容
- 当前状态 vs 预期结果
- 需要修改的相关文件
- 风险/注意事项（如适用）
- 适当的类型/优先级/工作量标签

## 如何实现

**提问**以填补信息空白 - 要简洁、尊重用户的上下文：
- 这个 issue/功能是什么
- 当前行为 vs 期望行为
- 类型（bug/feature/improvement）和优先级（如果还不明确）

保持问题简短。一条消息包含 2-3 个有针对性的问题。

**搜索上下文**仅在有帮助时：
- 查看相关文件以确认当前行为
- 确认它是否已存在于积压工作中
- 注意影响范围

**输出格式**

使用这个模板：

```markdown
# [清晰的标题]

## TL;DR
[一句话概述问题/功能]

## 当前 vs 预期
**当前**: [当前发生的情况]
**预期**: [应该发生的情况]

## 受影响的文件
- `path/to/file.ts`
- `path/to/other.ts`

## 类型 & 优先级
- 类型: [bug/feature/improvement]
- 优先级: [high/medium/low]
- 工作量: [small/medium/large]

## 注意事项
[可能影响的任何风险或其他区域]
```

创建后：
- 将其保存到项目的 `issues/` 目录
- 确认用户是否希望立即处理它
