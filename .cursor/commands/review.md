# Code Review Task

**目的**: 进行全面的代码审查。要彻底但保持建设性。

## 审查检查项

### **日志记录**
- 没有 console.log 语句，使用适当的日志记录器
- 日志级别适当（info, warn, error, debug）

### **错误处理**
- 对异步操作使用 try-catch
- 错误处理得当且信息丰富
- 中心化错误处理

### **TypeScript**
- 没有 `any` 类型，使用适当的接口/类型
- 类型定义清晰且准确
- 泛型使用得当

### **生产就绪性**
- 没有调试语句，没有注释掉的代码
- 没有硬编码的值，使用配置

### **React/Hooks**
- 副作用有清理，依赖项正确
- 没有无限重渲染的风险

### **性能**
- 没有不必要的重渲染，考虑使用 memo
- 昂贵的计算已优化
- 避免不必要的 API 调用

### **安全**
- 已检查授权，输入已验证
- 使用 RLS（行级安全）或适当的授权
- 敏感数据得到保护

### **架构**
- 遵循现有模式，代码结构良好
- 关注点分离清晰
- 可维护且可扩展

## 严重程度级别

- 🔴 **严重**: 必须在合并前修复（安全性、破坏性错误）
- 🟡 **中等**: 应该修复（性能、可维护性）
- 🔵 **轻微**: 可以考虑（代码风格、小优化）
- 💡 **建议**: 可选改进（最佳实践、未来增强）

## 审查原则

1. **建设性**: 提供解决方案，不只是指出问题
2. **具体**: 引用具体的代码行和文件
3. **优先级**: 关注高影响问题
4. **平衡**: 认可好的代码，不只是批评
5. **教育性**: 解释"为什么"，帮助开发者学习
