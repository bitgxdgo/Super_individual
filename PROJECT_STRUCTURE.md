# 项目结构说明

## 📂 完整文件树

```
Super_individual/
│
├── 📄 README.md                    # 项目主文档 - 从这里开始
├── 📄 QUICK_REFERENCE.md           # 快速参考指南 - 常用命令速查
├── 📄 PROJECT_STRUCTURE.md         # 本文件 - 项目结构说明
│
├── 🔧 .cursorrules                 # Cursor 编辑器配置
│
├── 📁 .cursor/                     # Cursor 命令配置目录
│   └── commands/                   # 所有工作流程命令
│       ├── README.md               # 命令系统完整文档
│       ├── create-issue.md         # 创建 Issue 命令
│       ├── explore.md              # 探索命令
│       ├── create-plan.md          # 创建计划命令
│       ├── execute.md              # 执行命令
│       ├── review.md               # 代码审查命令
│       ├── peer-review.md          # 跨模型审查命令
│       ├── document.md             # 文档更新命令
│       └── learning-opportunity.md # 学习模式命令
│
├── 📁 issues/                      # Issue 存储目录
│   └── .gitkeep                    # 保持目录存在
│
└── 📁 plans/                       # 执行计划存储目录
    └── .gitkeep                    # 保持目录存在
```

## 📖 文件说明

### 📄 主要文档

#### `README.md`
- **作用**: 项目主文档和入口
- **内容**: 
  - 项目概览
  - 快速开始指南
  - 使用示例
  - 最佳实践
- **适合**: 第一次了解这个工作流程

#### `QUICK_REFERENCE.md`
- **作用**: 命令速查表
- **内容**:
  - 所有命令的快速索引
  - 常用工作流程
  - 实用技巧
- **适合**: 日常开发参考

#### `PROJECT_STRUCTURE.md`
- **作用**: 项目结构说明（本文件）
- **内容**: 完整的文件树和说明
- **适合**: 了解项目组织结构

---

### 🔧 配置文件

#### `.cursorrules`
- **作用**: Cursor 编辑器的规则配置
- **内容**: 指向命令系统的引用
- **位置**: 项目根目录（隐藏文件）

---

### 📁 命令系统 (`.cursor/commands/`)

这是整个工作流程的核心！每个文件包含一个命令的完整提示词模板。

#### `create-issue.md` 
**用途**: 快速创建结构化的 Issue
- 包含标题、描述、优先级
- 自动识别相关文件
- 评估工作量和风险

**何时使用**: 
- 发现 bug
- 想到新功能
- 记录技术债务

---

#### `explore.md`
**用途**: 深入理解问题，不急于编码
- 分析现有代码库
- 识别依赖关系
- 列出不明确之处

**何时使用**:
- 开始任何新功能前
- 修复复杂 bug 前
- 重构前

**输出**: 探索摘要，包含问题理解、技术范围、需要澄清的内容

---

#### `create-plan.md`
**用途**: 创建详细的执行计划
- 分解为小步骤
- 状态跟踪（✅🟡⬜）
- 进度百分比

**何时使用**:
- 探索完成后
- 需要分阶段实施时
- 团队协作需要清晰计划

**输出**: Markdown 格式的执行计划，保存到 `plans/` 目录

---

#### `execute.md`
**用途**: 按计划逐步实施
- 遵循代码质量标准
- 更新进度跟踪
- 增量测试

**何时使用**:
- 计划批准后
- 开始实际编码

**特点**: 边实施边更新计划状态

---

#### `review.md`
**用途**: 全面的代码审查
- 检查日志、错误处理
- TypeScript 类型检查
- 性能和安全审查
- 架构合规性

**何时使用**:
- 功能完成后
- 合并前
- 定期代码质量检查

**输出**: 分类的问题列表（严重/中等/轻微/建议）

---

#### `peer-review.md`
**用途**: 跨模型代码审查
- 用其他 AI 模型审查
- 批判性评估反馈
- 验证每个发现

**何时使用**:
- 需要第二意见
- 复杂的架构决策
- 关键代码的额外验证

**特点**: 不盲目接受，要验证

---

#### `document.md`
**用途**: 更新所有相关文档
- 更新 CHANGELOG
- 更新 README
- 更新 API 文档
- 更新代码注释

**何时使用**:
- 任何代码更改后
- 添加新功能
- 修复 bug

**原则**: 文档应该反映代码，而不是代码应该做什么

---

#### `learning-opportunity.md`
**用途**: 深入学习新概念
- 三层次解释（2分钟/5分钟/10分钟）
- 80/20 原则
- 实用示例

**何时使用**:
- 遇到不熟悉的概念
- 需要深入理解某个技术
- 学习新的设计模式

**特点**: 从"为什么"开始，不只是"如何"

---

### 📁 工作目录

#### `issues/`
- **存储**: 使用 `/create-issue` 创建的 issue 文件
- **格式**: Markdown 文件
- **命名**: `[日期]-[issue-名称].md`

#### `plans/`
- **存储**: 使用 `/create-plan` 创建的执行计划
- **格式**: Markdown 文件，包含状态跟踪
- **命名**: `[功能名称]-plan.md`

---

## 🔄 文件之间的关系

```
1. 发现问题
   ↓
   /create-issue → issues/bug-xxx.md
   ↓
   
2. 理解问题
   ↓
   /explore → 探索摘要
   ↓
   
3. 制定计划
   ↓
   /create-plan → plans/feature-xxx-plan.md
   ↓
   
4. 执行实施
   ↓
   /execute → 更新 plans/feature-xxx-plan.md
   ↓
   
5. 代码审查
   ↓
   /review → 审查报告
   ↓
   (可选) /peer-review → 第二意见
   ↓
   
6. 更新文档
   ↓
   /document → 更新 README.md, CHANGELOG.md 等
```

## 📝 文件使用优先级

### 新用户（首次使用）
1. 📘 `README.md` - 了解整体概念
2. 📝 `QUICK_REFERENCE.md` - 快速上手
3. 📖 `.cursor/commands/README.md` - 详细了解命令
4. 🚀 开始使用第一个命令

### 日常开发
1. 📝 `QUICK_REFERENCE.md` - 查找命令
2. 📖 `.cursor/commands/[specific-command].md` - 查看命令详情
3. 📁 `plans/` - 跟踪进度
4. 📁 `issues/` - 管理待办

### 自定义配置
1. 🔧 `.cursorrules` - 调整 Cursor 配置
2. 📖 `.cursor/commands/[command].md` - 修改命令提示词
3. 📄 添加新的 `.md` 文件到 `commands/`
