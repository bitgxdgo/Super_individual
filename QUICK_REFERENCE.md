# 🚀 AI 开发工作流程 - 快速参考

## 📌 命令速查表

| 命令 | 用途 | 何时使用 |
|------|------|----------|
| `/create-issue` | 捕获 bug/功能 | 开发过程中想到新问题时 |
| `/explore` | 理解问题 | 开始编码前 |
| `/create-plan` | 制定计划 | 探索完成后 |
| `/execute` | 执行实施 | 计划批准后 |
| `/review` | 代码审查 | 实施完成后 |
| `/peer-review` | 跨模型审查 | 需要第二意见时 |
| `/document` | 更新文档 | 代码更改后 |
| `/learning-opportunity` | 学习模式 | 遇到不懂的概念时 |

---

## 🔄 标准工作流程

### 新功能开发
```
1. /explore          → 理解需求和现有代码
2. /create-plan      → 制定详细实施计划
3. /execute          → 按计划实施
4. /review           → 全面代码审查
5. /document         → 更新所有文档
```

### Bug 修复
```
1. /create-issue     → 记录 bug 详情
2. /explore          → 找到根本原因
3. /execute          → 修复问题
4. /review           → 验证修复
5. /document         → 更新 CHANGELOG
```

### 学习新技术
```
1. /learning-opportunity  → 三层次学习
2. 返回相关工作流程
```

## 💡 常用提示语

### 开始新功能
```
我想添加 [功能名称]。让我们从 /explore 开始，理解如何最好地集成它。
```

### 请求代码审查
```
我已经完成了 [功能]，请使用 /review 进行全面的代码审查。
```

### 创建执行计划
```
基于我们的探索，请使用 /create-plan 创建一个详细的实施计划。
```

### 更新文档
```
我刚完成了 [更改]，请使用 /document 更新所有相关文档。
```

## ⚡ 快捷技巧

### 上下文管理
- 📝 **上下文太长？** → 开始新会话，只带计划文件
- 🔄 **AI 重复犯错？** → 问它为什么会这样，然后更新文档

### 模型选择建议
- 🧠 **Claude** → 规划 & 复杂逻辑
- 🐛 **Codex** → 调试棘手 bug
- 🎨 **Gemini** → UI 开发
- ⚡ **Composer** → 快速迭代

### 质量检查点
- ✅ 开始前：用 `/explore` 确保理解
- ✅ 实施中：增量测试每个步骤
- ✅ 完成后：用 `/review` 检查质量
- ✅ 合并前：用 `/peer-review` 获取第二意见

---

## 🎯 最佳实践

### ✅ 做
- 先探索，再编码
- 保持计划文件更新
- 增量提交和测试
- 立即记录新 issue
- 代码变化后更新文档

### ❌ 不做
- 跳过探索阶段
- 偏离已批准的计划
- 等到最后才测试
- 忽略代码审查反馈
- 让文档过时
